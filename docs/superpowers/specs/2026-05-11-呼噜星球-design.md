## 1. 产品定位

### 1.1 一句话定义

> **呼噜星球** —— 和家人一起，把猫主子照顾得明明白白。

### 1.2 名字寓意

"呼噜" 是猫咪开心时发出的声音 —— 它是猫快乐的密语，是养猫人最熟悉的温度。"星球" 是治愈的容器 —— 每只猫是一颗星球，每个家庭是一个小星系。产品的核心使命是 **让每一颗呼噜星球持续转动**。

### 1.3 核心价值主张

1. **家庭协同** —— 解决"今天谁喂过了""疫苗谁带去打的"等多人合养的协同混乱
2. **可信内容** —— 用结构化 PGC 测评 + UGC 真实反馈，对抗选粮焦虑
3. **闭环服务** —— 不只是工具，把"找医院""找一只猫"也装进同一个产品

### 1.4 目标用户

| 优先级 | 画像 | 核心痛点 |
|--------|------|----------|
| **P0** | 城市 25-35 岁、家有 1-3 只猫、与伴侣/家人合养的女性 | 协同混乱 + 选粮焦虑 |
| **P1** | 单身养猫者、新手猫奴 | 科普诉求强、害怕做错 |
| **P2** | 有领养意向的潜在用户 | 不知道去哪找靠谱的小猫 |

### 1.5 差异化策略

| 维度 | 同类工具型产品 | 呼噜星球 |
|------|--------------|---------|
| 协同 | 单人记录 | 家庭多人协同 + 操作 timeline |
| 内容 | 文章列表 / 商品推荐 | 结构化 PGC 测评 + UGC 短评 |
| 服务 | 单一功能 | 记录 + 评测 + 医院 + 领养闭环 |
| 视觉 | 通用扁平 | 海盐白桃风（命名一致的强识别） |

## 2. 功能架构

### 2.1 模块概览

| 模块 | MVP 范围 | 明确不做（v1） |
|------|---------|--------------|
| 🐱 **家庭猫咪管理** | 家庭/成员管理；猫咪档案；进食/养护/健康记录；订阅消息提醒 | 体重智能曲线分析、AI 健康建议 |
| 🥫 **猫粮评测** | 猫粮 SKU 库；PGC 结构化测评；UGC 短评（带审核）；多维筛选 | 购物链接、比价、券 |
| 🏥 **附近医院** | 三方 POI 列表；自建评分评价层；导航跳转 | 在线挂号、问诊、付费咨询 |
| ❤️ **猫咪收养** | 运营录入待领养猫咪；用户提交领养申请；申请状态跟进 | 用户自主发布送养、机构入驻 |

### 2.2 一级导航（底部 4 Tab）

```
🪐 我的星球    🥫 猫粮评测    🐱 找猫咪    👤 我的
（首页/家庭）  （内容）       （医院+领养） （个人中心）
```

> **设计决策**：把"附近医院"与"猫咪收养"合并为"找猫咪" Tab，顶部分段切换。两者都是"找一只/找一个地方"的相似搜索场景，且都是低频功能（相对于每日记录），合并不损害体验，同时为未来可能新增的一级 Tab 留出位置。

### 2.3 二级页面树

```
🪐 我的星球
  ├── 家庭切换/邀请
```

## 3. 核心用户流程

### 3.1 首次使用 · 建立家庭

```
微信授权登录
  → 引导页（3 屏：家庭协同 / 养护记录 / 猫粮评测）
  → 创建你的呼噜星球（家庭名 / 默认"XX 的家"）
  → 添加第一只猫（头像 / 名字 / 性别 / 品种 / 生日 / 绝育 / 体重）
  → 引导邀请家人（生成邀请卡 / 可跳过）
  → 进入首页「我的星球」
```

### 3.2 日常喂食打卡

```
首页 → 快捷记录「喂食」
  → 选猫咪（多猫时；默认上次）
  → 选猫粮（家庭「在用」列表 / 临时输入）
  → 输入克数（预设 30/50/80g 快选）
  → 提交（自动记录时间 + 操作人）
  → 家庭其他成员看到 timeline 「XX 刚刚喂了 50g 主食」
```

### 3.3 疫苗提醒闭环

```
猫咪档案 → 健康 → 疫苗 → 添加疫苗记录
  → 填写（疫苗种类 / 打针日期 / 医院 / 费用）
  → 自动建议下次到期日（按疫苗类型）
  → 开启提醒 → 弹出微信订阅消息授权
  → 到期前 3 天 + 当天，推送家庭所有授权成员
  → 用户点推送进入小程序对应记录页 → 完成/延后/取消
```

### 3.4 评测 → 短评

```
评测 Tab → 猫粮详情
  → 阅读 PGC 测评（5 维度雷达图：原料 / 营养 / 性价比 / 适口性 / 安全性）
  → 看其他用户 UGC 短评
  → 点「我家猫吃过」 → 评分 + 短评（≤200 字）+ 1-3 张图
  → 短评进入待审核
  → 运营审核通过 → 显示在猫粮详情
```

### 3.5 领养申请

```
找猫咪 Tab → 等家的小猫 → 详情页
  → 查看猫咪信息 + 送养原因 + 所在城市
  → 点「我想领养它」 → 申请表
      （自我介绍 / 养猫经验 / 家庭环境 / 养宠承诺签署）
  → 提交 → 运营审核 → 对接送养方
  → 状态在「我的领养申请」可查（pending/approved/rejected/cancelled）
```

## 4. UI 设计规范

### 4.1 视觉风格

**海盐白桃风 · 软萌清新**。

软萌不等于过甜，目标人群覆盖男女通吃，因此采用**降低饱和度的"海盐桃"色板**而非饱满桃粉，以保证长时间界面使用不腻、男性用户也不抗拒。

### 4.2 色板

| 角色 | 色名 | HEX | 用途 |
|------|------|-----|------|
| 主色 | 浅雾桃 | `#FFDDD2` | 主按钮背景、卡片渐变、强调容器 |
| 副色 | 雾白 | `#FAF4ED` | 页面背景 |
| 强调色 | 烟桃红 | `#E58C7A` | 关键按钮、提醒红点、待办按钮 |
| 中性 1 | 海盐灰 | `#E8E0D5` | 分割线、不重要卡片背景 |
| 中性 2 | 深棕 | `#4A3F35` | 主要文字 |
| 中性 3 | 暖灰 | `#8A7B6E` | 辅助文字 |
| 中性 4 | 浅灰 | `#B0A294` | 极小标签 / placeholder |
| 点缀 | 薄盐蓝 | `#BFD7DE` | 中性提示标签、家庭成员区分色之一 |
| 白底 | 纯白 | `#FFFFFF` | 卡片底色 |

**色彩使用纪律**：
- 强调色 `#E58C7A` 只用于真正需要"行动"的位置（CTA 按钮、未读红点），不滥用
- 家庭成员区分色：浅雾桃 / 薄盐蓝 / 海盐灰 三色循环（最多 6 人家庭，颜色 + 字符首字组合区分）

### 4.3 字体

| 用途 | 字体 | 备注 |
|------|------|------|
| 中文 | `-apple-system, "PingFang SC", "HarmonyOS Sans", "AlibabaPuHuiTi-3"` | 系统字体优先链 |
| 数字 / 英文 | `"SF Pro Rounded", Inter, sans-serif` | 圆角数字字体强化软萌主调 |

字号体系：

| 用途 | px | weight |
|------|-----|--------|
| 页面标题 | 24 | 700 |
| 卡片标题 | 18-19 | 600-700 |
| 正文 | 14 | 400 |
| 辅助文字 | 12 | 400 |
| 极小标签 | 10-11 | 400-600 |

### 4.4 圆角与阴影

- **圆角**：所有元素圆角化，禁用直角
  - 小元素（标签、徽章）：`8px`
  - 中元素（按钮、输入框）：`14-16px`
  - 大元素（卡片）：`20-24px`
  - 头像：`50%`
- **阴影**：低饱和度低偏移，避免硬阴影
  - 卡片：`0 2px 12px rgba(74, 63, 53, 0.06)`
  - 主卡片：`0 4px 16px rgba(229, 140, 122, 0.12)`

### 4.5 Logo 设想

一颗圆形「呼噜星球」(海盐桃 `#FFDDD2`) + 上方蜷曲的奶油色猫剪影 + 几粒桃色光点漂浮。

### 4.6 动效原则

- 所有交互反馈用 `cubic-bezier(0.4, 0, 0.2, 1)` 缓动
- 卡片点击：缩放 `0.97`，140ms
- 页面切换：淡入 + 12px 上移，220ms
- 不使用弹性回弹（避免过度活泼）

### 5.3 数据隐私规则

- 家庭数据（猫咪 / 记录 / 提醒）**仅限家庭成员可见**，后端按 `family_id` 强隔离
- UGC 短评、医院评价、领养申请的对外内容均需**运营审核**才公开
- 图片走 OSS 私有桶 + 签名 URL，禁止公开桶
- 用户注销时硬删除用户表数据，关联记录脱敏保留（用于猫咪历史完整性）

### 6.4 API 设计原则

- RESTful 风格，资源化命名（`/api/v1/cats/{id}/feedings`）
- 统一响应包装：`{ code, message, data, traceId }`
- 鉴权：HTTP Header `Authorization: Bearer <jwt>`（JWT 内含 `user_id`）
- 家庭隔离：HTTP Header `X-Family-Id` 传当前家庭 ID；服务端拦截器校验该用户是否为该家庭成员 + 服务层二次校验资源归属
- 列表接口统一分页参数：`page` `size` `cursor`（按需）

### 6.5 关键安全控制

| 风险 | 控制 |
|------|------|
| 家庭越权访问 | 后端拦截器 + 服务层二次校验 family_id |
| 短评/评价违规内容 | 100% 人工审核 + 关键词预筛 + 用户举报 |
| 图片违规 | 上传时调用微信内容安全检测接口 |
| 接口刷量 | Nginx 限流 + Redis 滑动窗口（按 user_id） |
| 个人信息泄漏 | OSS 签名 URL（有效期 1h）+ 数据库手机号加密存储 |

## 8. 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| **订阅消息授权率低** | 提醒到达差，核心场景失效 | 在关键节点引导授权；提供"加入手机日历"作为备选；不强依赖单一通道 |
| **内容冷启动** | 评测/医院评价/待领养数据稀疏，留存差 | 上线门槛硬约束：≥30 PGC / ≥100 SKU / ≥20 待领养；上线后周更节奏（运营 SOP） |
| **家庭权限边界 bug** | 数据越权，严重事故 | API 层拦截器 + 单元/集成测试覆盖；不依赖前端控制 |
| **微信类目审核驳回** | 上线延期 | 避开"医疗服务"类目；定位为"养宠工具"；收养信息明确标注"信息聚合，非中介" |
| **多猫家庭体验割裂** | 多猫用户操作繁琐 | 全局当前猫咪上下文；每次快捷记录默认上一只；多猫页内大头像快速切换 |
| **三方 POI 数据噪声** | 医院列表脏 | 用户举报通道；运营每周清洗一次；保留本地覆盖字段 |
| **UGC 内容审核成本** | 运营负担过重 | 200 字短评限制 + 图片数量限制；关键词预过滤；先做"先审后发"，量级上来再考虑"先发后审" |
